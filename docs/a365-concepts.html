The conceptual model behind Microsoft Agent 365

What is Agent 365 (A365)?

Microsoft Agent 365 is Microsoft's IT admin control plane for AI agents. It gives organisations real-time visibility into their agentic environment, a centralised registry of all agents, and the ability to apply identity, security, governance, and lifecycle management controls at scale.

Every agent registered in A365 is defined by a blueprint, which is essentially the template from which agent instances are created. Every A365 agent is a point in a small grid: own identity (yes/no) × auth mode (OBO / S2S / both). The choice of grid cell determines what shows up in Entra, who consents, how spans get attributed, and what your audit story looks like. This page maps that grid, walks you to the right cell for your scenario, and lets you click through every concept in the stack — 85 of them, with animated token flows for the two most common modes.

Section 2

Two fundamental axes

Every A365 registration class can be placed on a 2×3 grid. The own-identity axis determines whether the agent gets its own Entra agentic user (AI Teammate) or just a service principal (Blueprint-only). The auth-mode axis determines how it acquires tokens at runtime. Hover any cell to see the corresponding class.

Axis 1: Own Identity (Y axis) — AI Teammate vs Blueprint-only

This is the big one. It determines whether your agent gets its own full Entra identity or just a service principal.

AI Teammate (top row)

An AI teammate operates in Microsoft 365 using its own agent identity. When you create an agent identity blueprint, the Agent 365 CLI provisions a Microsoft Entra Agent ID for your agent, which is a first-class identity in your tenant, subject to the same enterprise governance policies as human users.

In plain terms: the agent gets its own user account. Agentic users can be assigned licences like Microsoft 365 E5 or Teams Enterprise, have their own mailbox and OneDrive storage, appear in the org chart, and can be @mentioned in Teams and documents. Your agent can literally send emails from its own mailbox, show up in org charts, and be treated like a colleague. That's what "AI Teammate" means.

Blueprint-only (bottom row)

No dedicated user account. Agents (not AI teammates) don't need a config file. Use a365 setup all --agent-name <name> and the CLI resolves your tenant and client app automatically. You get a service principal in Entra, but no M365 user identity. Simpler, less overhead, less licensing cost, and it's the default --m365 mode.

Axis 2: Auth Mode (X axis) — How the agent gets tokens at runtime

OBO / Delegated (left column)

On-Behalf-Of. This flow enables an agent to act on behalf of a user. The agent receives a user's delegated token, exchanges it to perform actions as if the user is performing them, and operates with the user's permissions and context. Classic scenario: a user triggers the agent, and the agent does things as that user. Access is scoped to what the triggering user is allowed to do.

S2S / App Permissions (middle column)

Service-to-Service. The agent authenticates using its own credentials (agent blueprint credentials), operates independently with its own assigned permissions, and has its own identity separate from any user. This flow is ideal for autonomous agent operations such as scheduled tasks, monitoring, sending emails from the agent's mailbox, or background processing without user interaction. No user needs to be present. The agent runs on its own authority.

Both (right column)

The agent can do either. Flexible, but also more complex to govern.

Same model in plain English: the school analogy

Think of it like a school robotics programme.

  • The Blueprint is the textbook — a template anyone can use to learn how to build a particular kind of robot.
  • When the school buys the textbook (blueprint added to tenant), it ends up in your school as the Blueprint Principal — your school's copy, holding your school's credentials, and able to build robots for your students.
  • Each robot the Blueprint Principal builds is an Agent Identity. Same template, individual robot. Has a name badge, no PIN code (it authenticates via secure tokens rather than a password it could leak on the playground).
  • Some robots get an extra Agent User facet — a school mailbox, a OneDrive locker, and a row in the class register. Robots that need to email teachers as themselves.
  • Every robot has a Human Sponsor — a teacher accountable for what the robot does. If the teacher leaves the school, Lifecycle Workflows reassign the robot (or retire it) automatically.

The technical pattern matches multi-tenant App Registration → Service Principal — if your team already understands that pattern, they understand 80% of A365 identity. Lead with the analogy when explaining to architects new to A365.

Section 2.5

Where agents come from

Five creation paths converge into A365. Microsoft-platform agents (Copilot Studio, Agent Builder, Foundry, A365 SDK) auto-register. External-platform agents (SAP, ServiceNow, Slack) manual-register via Graph API. Anything else is a shadow agent — invisible to A365's identity, governance, and security planes.

External agents (manual-register) — what it actually takes to register + get observability

External-platform agents (SAP Joule, ServiceNow Now Assist, Slack workflow bots, AWS Bedrock agents, etc.) don't auto-register. There's a deliberate per-agent registration flow via Microsoft Graph plus an observability-instrumentation contract that the platform owner has to implement. Two paths exist:

Path 1 — Registry-only (metadata-only visibility)

Cheapest path. The agent appears in the M365 Admin Centre Agents inventory, has owner + sponsor metadata, sits in a collection, and is visible in the Agent Map — but no token-level Conditional Access enforcement, no DLP on the agent's interactions, no Defender runtime inspection. Useful for inventory + sponsor accountability, not for governance enforcement.

Path 2 — Full identity integration (CA + DLP + Defender)

Full path. The external agent gets an Entra App Registration + Service Principal, joins the Agent Identity space, and is subject to every governance primitive: Conditional Access targeting, sensitivity labels, DLP, Defender for Cloud Apps runtime inspection, Sentinel correlation, observability pipeline.

Registration mechanism — what gets done by whom

External platform owner (e.g. SAP / ServiceNow team)

  1. Create or designate an Entra App Registration for the agent in the customer's tenant (or use a multi-tenant app of theirs that the customer admin-consents into).
  2. Set the App Registration's OData type to agent via Graph PATCH on /applications/{id}.
  3. Provide an Agent Card Manifest describing the agent's skills, capabilities, and discovery info — registered via POST /agents on Microsoft Graph.
  4. If observability is required (Path 2): instrument the agent code with the A365 Observability SDK (Python, .NET, JS available) or implement the structured-scope OTel contract directly (InvokeAgentScope / InferenceScope / OutputScope / ExecuteToolScope) and POST spans to agent365.svc.cloud.microsoft/observability/....
  5. Acquire Agent365.Observability.OtelWrite tokens via client_credentials (S2S) using the agent's own clientId/secret, or via OBO if the agent is invoked with a calling-user JWT.

A365 admin (customer tenant)

  1. Admin-consent the external platform's app permissions (the OData-type-agent app needs tenant-wide consent to call Graph + the Observability API).
  2. Register the agent's metadata in the Agent Registry via POST /agents (or via M365 Admin Centre → Agents → "Register external agent" if the UI is available).
  3. Assign Owner (technical contact at the customer or at the platform vendor) and Sponsor (business-accountable person on the customer side).
  4. Place the agent in a Collection — typically Quarantine first, promote to a Custom or Global collection after vetting.
  5. Configure Conditional Access policies targeting the agent's SP (require MFA on inbound, block from risky locations, etc.).
  6. Configure Purview DLP + sensitivity labels to apply to the agent's interactions.

Security team

  1. Review the app permissions requested — flag over-broad scopes for negotiation.
  2. Add the agent's SP to ID Protection monitoring (anomalous sign-in alerts).
  3. Author Sentinel detection rules on CloudAppEvents filtered to the agent's SP.
  4. If sensitive data is in scope: run Security Exposure Management against the agent to map blast radius before promoting out of Quarantine.

Sponsor (business)

  1. Acknowledge accountability for the agent's actions in the directory record.
  2. Sign off on the use case, data scope, and user audience.
  3. Nominate a successor for Lifecycle Workflows auto-succession on departure.

Observability — concretely, what the external agent's code has to do

For Path 2, the agent code must:

  1. On every turn, wrap the agent's processing in an InvokeAgentScope (outer scope, holds tenant_id + agent_id + user_id baggage).
  2. Wrap each LLM call in an InferenceScope nested inside InvokeAgentScope.
  3. Wrap each tool / function / MCP call in an ExecuteToolScope.
  4. Wrap the response emission in an OutputScope.
  5. Configure the OTel exporter to POST to https://agent365.svc.cloud.microsoft/observability/tenants/{tenant_id}/otlp/agents/{agent_id}/traces with a bearer token scoped to api://9b975845-388f-4429-889e-eab1ef63949c/Agent365.Observability.OtelWrite.
  6. Set gen_ai.operation.name on every span to one of {chat, invoke_agent, execute_tool, output_messages} — the A365 ingest endpoint rejects anything else.

The A365 SDK (Python / .NET / JS) does all of this automatically if you use it. If your platform isn't supported by the SDK, you implement the OTel contract directly — the wire format is plain OTLP HTTP with a Bearer token.

Sources. Microsoft Graph /agents endpoint + OData type agent are public-but-evolving APIs; verify against Register agents to the registry for the current Graph schema.
